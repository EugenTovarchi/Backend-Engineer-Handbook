## Исходная ситуация

Браузер уже разобрал URL и получил IP-адрес через DNS:

```text
company.com -> 203.0.113.17
```

Теперь он знает, к какой машине нужно идти.

Но на одной машине может работать много программ:

```text
Web API
PostgreSQL
Redis
RabbitMQ
SSH
Monitoring
```

IP-адрес отвечает только на вопрос:

> К какой машине подключаться?

Но остаётся следующий вопрос:

> К какому приложению внутри этой машины нужно попасть?

Для этого нужен port.

---

## Зачем нужна эта глава

Port (порт — числовая сетевая точка входа приложения) — один из самых практичных сетевых терминов для backend-разработчика.

Он встречается постоянно:

| Где | Пример |
|---|---|
| ASP.NET Core | `http://localhost:5000` |
| Docker Compose | `8080:8080` |
| PostgreSQL | `5432` |
| Redis | `6379` |
| RabbitMQ | `5672`, `15672` |
| HTTPS | `443` |
| HTTP | `80` |
| Nginx | `listen 80;`, `proxy_pass http://service:8080` |

Если не понимать порты, сложно разбираться в ошибках:

- приложение запущено, но не открывается;
- контейнер работает, но с host machine к нему не подключиться;
- Nginx возвращает `502 Bad Gateway`;
- сервис пытается подключиться к `localhost:5432`, но PostgreSQL там нет;
- в Docker Compose перепутаны внутренний и внешний порты.

## Короткое определение

**Port (порт — числовая сетевая точка входа приложения)** — это числовой идентификатор сетевого приложения внутри машины.

Упрощённо:

```text
IP   = адрес здания
Port = дверь в конкретный кабинет
```

Пример:

```text
203.0.113.17:443
```

означает:

```text
подключиться к машине 203.0.113.17 на порт 443
```

---

## Простое объяснение

Представь бизнес-центр.

Адрес здания:

```text
ул. Примерная, дом 10
```

Но в здании много офисов.

Чтобы попасть в нужную компанию, недостаточно знать адрес здания. Нужно знать кабинет.

В сети:

```text
IP   = адрес здания
Port = кабинет / дверь / вход
```

Поэтому почти любое подключение выглядит как пара:

```text
host + port
```

Например:

```text
company.com:443
postgres:5432
redis:6379
rabbitmq:5672
```

---

## Известные порты

Некоторые порты используются настолько часто, что стали стандартными.

| Port | Обычно используется для |
|---:|---|
| `80` | HTTP |
| `443` | HTTPS |
| `5432` | PostgreSQL |
| `6379` | Redis |
| `5672` | RabbitMQ AMQP |
| `15672` | RabbitMQ Management UI |
| `22` | SSH |
| `5000`, `5001`, `8080` | часто используются в dev/backend окружениях |

Важно: порт сам по себе не гарантирует протокол.

Например, можно запустить HTTP-сервер на `9000`, если приложение так настроено.

Но есть общепринятые соглашения.

---

## Порт по умолчанию

В URL порт может быть не указан явно.

Например:

```text
https://company.com/api/files/123
```

Но браузер понимает:

```text
https -> 443
```

То есть фактически:

```text
https://company.com:443/api/files/123
```

А для HTTP:

```text
http://company.com/api/files/123
```

обычно означает:

```text
http://company.com:80/api/files/123
```

---

## Внутренний и внешний порт в Docker

В Docker Compose часто встречается запись:

```yaml
ports:
  - "8080:8080"
```

Или:

```yaml
ports:
  - "5000:8080"
```

Формат:

```text
HOST_PORT:CONTAINER_PORT
```

Пример:

```yaml
ports:
  - "5000:8080"
```

означает:

```text
на host machine открыть порт 5000
и пробросить его в контейнер на порт 8080
```

То есть снаружи:

```text
http://localhost:5000
```

а внутри контейнера приложение слушает:

```text
http://0.0.0.0:8080
```

Это частая точка ошибок.

---

## Port и Kestrel

ASP.NET Core приложение принимает HTTP-запросы через Kestrel (встроенный web server ASP.NET Core — компонент, который принимает HTTP-запросы и передаёт их в ASP.NET Core pipeline).

Kestrel должен слушать конкретный endpoint:

```text
http://0.0.0.0:8080
```

или:

```text
https://localhost:5001
```

В Docker часто указывают:

```yaml
environment:
  ASPNETCORE_URLS: http://+:8080
```

Смысл:

```text
слушать входящие HTTP-подключения на порту 8080
```

Если Nginx проксирует в сервис на порт `8080`, а приложение слушает `5000`, запрос не дойдёт.

---

## Port и Nginx

Nginx тоже слушает порты.

Пример:

```nginx
server {
    listen 80;

    location /directory-service/ {
        proxy_pass http://directory-service:8080;
    }
}
```

Здесь есть два разных порта:

```text
listen 80
```

— порт, на котором Nginx принимает внешние запросы.

```text
http://directory-service:8080
```

— порт, на который Nginx отправляет запрос внутри Docker-сети.

Это разные уровни.

---

## Практика из проекта

В локальной инфраструктуре может быть единая точка входа:

```text
http://localhost:8080
```

Клиент обращается к Nginx на порт `8080` host machine.

Дальше Nginx по path-prefix отправляет запрос в нужный сервис:

```text
/directory-service/... -> directory-service:8080
/file-service/...      -> file-service:8080
/auth-service/...      -> auth-service:8080
```

То есть внешний порт может быть один, а внутренних сервисов несколько.

Именно Reverse Proxy (обратный прокси — сервер, который принимает запрос клиента и передаёт его внутреннему сервису) решает, куда направить запрос.

## Типичные ошибки

### Ошибка 1. Путать host port и container port

```yaml
ports:
  - "5000:8080"
```

`5000` — порт на host machine.

`8080` — порт внутри контейнера.

---

### Ошибка 2. Думать, что если контейнер запущен, сервис точно доступен

Контейнер может работать, но приложение внутри может слушать другой порт.

Или порт не проброшен наружу.

---

### Ошибка 3. Проксировать Nginx на неправильный port

Если приложение слушает `8080`, а Nginx отправляет на `5000`, будет ошибка соединения.

---

### Ошибка 4. Думать, что порт определяет протокол жёстко

`443` обычно HTTPS, но технически приложение можно настроить иначе.

Порт — это соглашение и точка входа, а не магическая гарантия протокола.

## Ответ для собеседования

Port — это числовой идентификатор сетевого приложения на машине. IP помогает найти узел, а port указывает, к какому приложению на этом узле подключаться. Для backend-разработчика это важно при работе с Kestrel, Nginx, Docker Compose, PostgreSQL, Redis и RabbitMQ. Например, `https` обычно использует `443`, PostgreSQL — `5432`, Redis — `6379`. В Docker важно отличать host port от container port: запись `5000:8080` означает, что снаружи используется `5000`, а внутри контейнера приложение слушает `8080`.

---

## Шпаргалка

- IP отвечает на вопрос: к какой машине идти.
- Port отвечает на вопрос: к какому приложению подключаться.
- `80` — обычно HTTP.
- `443` — обычно HTTPS.
- `5432` — PostgreSQL.
- `6379` — Redis.
- `5672` — RabbitMQ AMQP.
- В Docker `HOST_PORT:CONTAINER_PORT` — это разные порты.
- Kestrel должен слушать тот port, на который обратный прокси отправляет запрос.
- Nginx может слушать один внешний port и проксировать на разные внутренние сервисы.
