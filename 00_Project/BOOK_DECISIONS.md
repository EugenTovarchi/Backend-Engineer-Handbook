# Book Decisions

Журнал архитектурных и редакционных решений книги `Backend Engineer Handbook`.

Назначение документа:

> Объяснить, почему книга устроена именно так.

---

# Decision 001

Статус

Accepted

Дата

2026-07-11

Решение

Главный принцип книги:

> Не запоминай технологии.
> Понимай, какие проблемы они решают.

Причина

Книга должна помогать backend-разработчику понимать инженерные решения, а не заучивать определения без контекста.

Последствия

Каждая глава начинается с проблемы, которую решает технология, и связывает тему с реальной работой .NET backend-разработчика.

---

# Decision 002

Статус

Accepted

Дата

2026-07-11

Решение

Использовать путешествие одного запроса как фундамент книги.

Причина

Путь запроса естественно связывает сеть, входной слой, ASP.NET Core, бизнес-логику, данные, интеграции, наблюдаемость и production-доставку.

Последствия

Темы добавляются не как независимый справочник, а как ответы на вопросы, возникающие по ходу движения запроса через backend-систему.

---

# Decision 003

Статус

Review

Дата

2026-07-11

Решение

Использовать модульную структуру книги.

Причина

Модули помогают вести книгу итерациями, завершать крупные смысловые блоки и выполнять редакторский проход после каждого блока.

Последствия

Модульная структура проверяется на первом модуле. Окончательное решение о формате модулей будет принято после пользовательского review.

---

# Decision 004

Статус

Accepted

Дата

2026-07-11

Решение

GitHub является единственным источником истины.

Причина

GitHub хранит актуальное состояние книги, историю изменений и позволяет продолжать работу из нового чата или новой CLI-сессии без потери контекста.

Последствия

Все существенные изменения фиксируются commit-ами и push-атся в основной репозиторий. Локальные заметки не считаются актуальным состоянием книги, пока не попали в GitHub.

---

# Decision 005

Статус

Accepted

Дата

2026-07-11

Решение

Использовать один репозиторий.

Причина

Один репозиторий снижает риск расхождения версий, упрощает навигацию и сохраняет единый контекст для глав, проектных документов и редакционных решений.

Последствия

Не создаются отдельные репозитории для модулей, черновиков или инфраструктурных документов книги.

---

# Decision 006

Статус

Accepted

Дата

2026-07-11

Решение

Использовать Markdown Links вместо Obsidian Links.

Причина

Markdown-ссылки работают и в GitHub, и в Obsidian preview. Obsidian wiki-links хуже подходят для публичного репозитория и будущего экспорта.

Последствия

Ссылки оформляются в формате `[Название](./path/to/file.md)`. Obsidian wiki-links не используются в главах книги.

# Decision 007

Статус

Accepted

Дата

2026-07-11

Решение

Первое появление английского термина в главе оформлять так:

```text
English Term (русский перевод — краткое объяснение)
```

Причина

Такой формат сохраняет английский термин для интервью и документации, но сразу делает его понятным для читателя.

Последствия

После первого объяснения в пределах главы можно использовать короткую форму термина.

# Decision 008

Статус

Accepted

Дата

2026-07-11

Решение

Не приписывать автору или проекту опыт без подтверждения.

Причина

Книга должна быть технически честной. Нельзя писать `в твоём проекте использовалось`, если это не подтверждено исходными материалами или существующими главами.

Последствия

Используются формулировки `может использоваться`, `часто применяется`, `в backend-проектах встречается`, если нет подтверждённого проектного факта.

---

# Decision 009

Статус

Accepted

Дата

2026-07-11

Решение

Не объяснять термин полностью повторно, если он уже подробно разобран в предыдущей главе. Использовать короткое напоминание и ссылку.

Причина

Повторы раздувают главы и мешают последовательному чтению.

Последствия

Следующие главы ссылаются на предыдущие через Markdown-ссылки и дают только минимальное напоминание.

# Decision 010

Статус

Accepted

Дата

2026-07-11

Решение

Не углубляться дальше потребностей Middle .NET Backend-разработчика.

Причина

Книга готовит к backend-интервью и реальной работе, а не заменяет учебник по сетевой инженерии, криптографии, DevOps или computer science.

Последствия

Глубокие детали допускаются только тогда, когда они помогают понять практическое поведение ASP.NET Core, Kestrel, Nginx, Docker, Web API или production-диагностику.

# Decision 013

Статус

Accepted

Дата

2026-07-11

Решение

Один модуль ведётся как отдельный рабочий цикл:

```text
один модуль
  ↓

  ↓
новая CLI-сессия
```

Причина

Последствия

Перед началом нового модуля нужно читать `STYLE_GUIDE.md`, `BOOK_DECISIONS.md`, `ROADMAP.md`, `PROJECT_STATUS.md` и `EDITOR_HANDOFF.md`.

# Decision 015

Статус

Accepted

Дата

2026-07-11

Решение

Codex CLI не должен по собственной инициативе изменять документы управления проектом:

- `00_Project/STYLE_GUIDE.md`
- `00_Project/BOOK_DECISIONS.md`

Последствия

Codex CLI может читать эти документы и применять содержащиеся в них правила, но не должен менять их без отдельного задания.

# Decision 016

Статус

Accepted

Дата

2026-07-11

Решение

Модуль II посвящён ASP.NET Core Request Pipeline. Production Entry Layer выделяется в отдельный более поздний модуль. Полноценная authentication-система выделяется в отдельный модуль сразу после Request Pipeline.

Authentication и authorization в Модуле II ограничены их ролью в pipeline.

JWT, access token, refresh token, OAuth 2.0, OpenID Connect, ASP.NET Core Identity, хранение пользователей и AuthService обязательно входят в книгу и раскрываются в отдельном Модуле III.

Причина

Нужно не смешивать три разные инженерные истории:

1. обработка запроса внутри ASP.NET Core;
2. установление личности и управление авторизацией;
3. production-инфраструктура перед приложением.

Последствия

Ближайшая последовательность модулей фиксируется так:

1. Модуль II — ASP.NET Core Request Pipeline;
2. Модуль III — Аутентификация и авторизация;
3. Модуль IV — Production Entry Layer.

Модуль II объясняет, где authentication и authorization находятся в pipeline, но не превращается в модуль про JWT, refresh token, OAuth/OIDC, ASP.NET Core Identity или архитектуру AuthService.

# Decision 017

Статус

Accepted

Дата

2026-07-11

Решение

ASP.NET Core Request Pipeline не заменяет остальные темы ASP.NET Core.

Потерянный при перестройке scope восстанавливается отдельным Модулем V `ASP.NET Core Web API и инфраструктура приложения`.

Модуль V будет посвящён Web API execution и инфраструктуре приложения.

Controllers, Minimal APIs, Filters, Model Binding, Validation, ProblemDetails, Dependency Injection, Configuration, Hosted Services, Health Checks, API Versioning, OpenAPI и Rate Limiting остаются обязательной частью книги.

Текущая работа не создаёт главы Модуля V.

CORS остаётся в Review до утверждения структуры Модуля V.

Причина

Request Pipeline объясняет движение `HttpContext` до endpoint, но не раскрывает внутренний Web API execution, связывание входных данных, валидацию, DI, конфигурацию и остальные стандартные процессы ASP.NET Core приложения.

Последствия

Модуль II остаётся сфокусированным на pipeline от Kestrel до endpoint. Web API execution и инфраструктурные возможности ASP.NET Core будут разобраны позже отдельным модулем, без потери этих тем из roadmap.

# Decision 018

Статус

Accepted

Дата

2026-07-13

Решение

Утвердить окончательное название Модуля III:

```text
Модуль III. Аутентификация и авторизация в ASP.NET Core: Cookies, JWT, OAuth 2.0 и OpenID Connect
```

Утвердить маршрут Модуля III из 17 глав:

1. Кто обращается к системе: Identity, Authentication и Authorization
2. Учётная запись, credentials и хранение пользователей
3. Парольная аутентификация и безопасное хранение паролей
4. Модель Authentication в ASP.NET Core
5. Cookie Authentication и аутентифицированная session
6. Access Token и Bearer Authentication
7. JWT и проверка token
8. Refresh Token и жизненный цикл token
9. Claims, Roles и Permissions
10. Policy-based и Resource-based Authorization в ASP.NET Core
11. OAuth 2.0: делегирование доступа, роли и scopes
12. Authorization Code Flow и PKCE
13. OpenID Connect и внешние Identity Providers
14. ASP.NET Core Identity
15. OpenIddict
16. Архитектура AuthService и границы distributed system
17. Полный путь аутентификации и авторизации

Модуль III не строится вокруг ложной модели `Authentication = JWT`. Cookie Authentication и token-based authentication рассматриваются как разные модели. Password Hashing, Cookie Authentication, Access Token, JWT validation и Refresh Token lifecycle разделены на самостоятельные главы, потому что они отвечают на разные инженерные вопросы.

Claims, Roles и Permissions отделяются от Policies, Requirements и Handlers. OAuth 2.0 и OpenID Connect не смешиваются: OAuth 2.0 объясняется как делегирование доступа, а OpenID Connect — как идентификация поверх OAuth 2.0. Authorization Code Flow + PKCE раскрывается отдельной главой.

ASP.NET Core Identity, OpenIddict и AuthService не считаются одним framework:

- ASP.NET Core Identity отвечает за управление пользователями, паролями, ролями, claims и связанными сущностями;
- OpenIddict является .NET-стеком для OAuth 2.0 / OpenID Connect client, server и token validation scenarios;
- AuthService является архитектурным вариантом выделения границ authentication/authorization в distributed system, а не обязательным сервисом для каждого проекта.

Security threats и типичные ошибки не получают отдельную изолированную главу. Они раскрываются рядом с соответствующими механизмами и сводятся в итоговой главе.

Границы Модуля III:

- не дублировать Модуль II, где authentication и authorization объяснены как этапы request pipeline;
- не дублировать Модуль IV, где будет production entry layer;
- не дублировать Модуль V, где будет Web API execution и общая инфраструктура приложения;
- не переносить CORS в Модуль III автоматически;
- раскрывать CSRF там, где он относится к Cookie Authentication и redirect-based flows.

Причина

Authentication и authorization часто ошибочно сводят к JWT, из-за чего теряется различие между установлением identity, хранением credentials, cookie-session моделью, bearer-token моделью, claims-моделью, authorization policies, OAuth 2.0, OpenID Connect и инфраструктурными framework.

Разделение этих тем на отдельные главы позволяет объяснить authentication-систему как инженерную цепочку решений, а не как набор несвязанных библиотек и token-форматов.

Последствия

Модуль III начинается только после отдельного задания на создание структуры и глав. Текущий Decision фиксирует архитектуру и маршрут, но не создаёт главы, папки, схемы или черновики.
