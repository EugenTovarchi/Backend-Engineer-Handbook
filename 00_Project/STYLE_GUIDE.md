# Style Guide

Этот файл фиксирует единый стиль будущей книги.

## Цель книги

Не просто собрать определения, а объяснять инженерные решения так, чтобы материал можно было использовать:

- на собеседовании Middle+/Senior;
- в реальной работе;
- для повторения перед интервью;
- как личную инженерную базу знаний.

## Структура каждой главы

Каждая полноценная глава должна иметь следующий формат.

### 1. Проблема

Сначала объясняем, какую инженерную проблему решает технология.

Не начинать с сухого определения, если можно начать с ситуации из практики.

### 2. Зачем нужна эта глава

Используем именно формулировку `Зачем нужна эта глава`, а не `Зачем появилась эта глава`.

Здесь объясняем, почему backend-разработчику важно понимать тему и где она встретится в следующих главах или в работе.

### 3. Эта глава понадобится позже

Добавляем Obsidian-ссылки на связанные будущие главы.

Пример:

```md
[[Kestrel]]
[[Nginx]]
[[Docker Networking]]
[[Reverse Proxy]]
```

### 4. Короткое определение

2–5 предложений, которые можно сказать на собеседовании.

### 5. Простая аналогия

Аналогия нужна для понимания, но не должна заменять техническое объяснение.

Пример:

- Nginx — секретарь/ресепшен, который принимает все звонки и направляет их в нужный отдел.
- SemaphoreSlim — парковка с ограниченным количеством мест.
- RabbitMQ — почтовое отделение/очередь сообщений.
- Redis — быстрый стол рядом с рабочим местом, а не дальний склад.

### 6. Техническое объяснение

Здесь объясняем реальные термины:

- протоколы;
- жизненный цикл;
- потоки;
- память;
- транзакции;
- конфигурации;
- ограничения.

### 7. Схема

Схемы используются только там, где они реально улучшают понимание.

Для простых потоков и последовательностей используем Mermaid:

```mermaid
flowchart LR
```

или

```mermaid
sequenceDiagram
```

Для крупных архитектурных схем используем SVG, потому что он хорошо поддерживается GitHub, Obsidian, браузерами и PDF-экспортом.

### 8. Пример из проекта

Если тема применялась в проектах, обязательно добавляем практический блок:

- где использовалось;
- зачем;
- какой файл/класс/компонент;
- какую проблему решало;
- какие были ограничения.

### 9. Мини-пример кода или конфигурации

Код должен быть коротким и объясняющим идею.

Не вставлять большие куски без необходимости.

### 10. Типичные ошибки

Формат:

```text
Ошибка: ...
Почему неверно: ...
Как правильно: ...
```

### 11. Вопросы собеседования

Делить по уровню:

- Junior
- Middle
- Senior
- Architect / System Design

Ответы на вопросы по умолчанию делаем свернутыми через HTML `<details>`, чтобы глава не превращалась в длинную простыню.

Пример:

```md
<details>
<summary>Ответ</summary>

Краткий, но технически точный ответ.

</details>
```

Такой формат поддерживается GitHub и нормально читается в Obsidian preview.

### 12. Ответ для собеседования

Готовый ответ на 60–120 секунд.

Не зубрёжка, а инженерная формулировка.

### 13. Шпаргалка

5–10 пунктов, которые нужно помнить.

## Правила качества

1. Не сглаживать неточности.
2. Не писать утверждения, если нет уверенности.
3. Отделять факт от интерпретации.
4. Показывать ограничения технологии.
5. Не изображать личный опыт там, где его не было.
6. Если пример взят из проекта — формулировать честно, что именно делалось.
7. Если тема сложная — сначала объяснить простыми словами, потом технически.
8. Не превращать книгу в пересказ документации.
9. Не делать главы изолированными — показывать связи с другими темами.
10. Не оставлять спорные формулировки без уточнения.

## Стандарт схем

### Mermaid

Mermaid используем для:

- простых flowchart-схем;
- sequence diagram;
- коротких схем обработки запроса;
- быстрых поясняющих диаграмм внутри текста.

Mermaid не используем для больших архитектурных схем, где важны точное выравнивание, крупные иконки, цветовые зоны и визуальная иерархия.

### SVG

SVG используем для главных архитектурных схем главы.

Требования к SVG-схемам:

1. Иконки должны быть крупными и читаемыми.
2. Основные блоки должны быть выровнены по сетке.
3. Центральная ось схемы должна быть визуально ровной.
4. Отступы между блоками должны быть одинаковыми.
5. Стрелки не должны пересекаться без необходимости.
6. Подписи должны быть читаемы в GitHub и Obsidian без увеличения.
7. Цвета должны помогать группировать смысл, а не украшать схему.
8. Если схема выглядит криво или перегруженно — переделываем, а не оставляем.

Главная схема главы должна выглядеть как инженерная иллюстрация, а не как черновик.

## Формат Obsidian-ссылок

Для связей между главами использовать внутренние ссылки:

```md
[[Nginx]]
[[Docker]]
[[Kestrel]]
[[Reverse Proxy]]
[[Load Balancing]]
```

## Термины

Термины на английском сохраняем, если они общеупотребимы в интервью:

- Reverse Proxy
- Load Balancer
- Middleware
- ThreadPool
- CancellationToken
- Expression Tree
- Outbox Pattern
- Read Replica

Русское объяснение даём рядом.
