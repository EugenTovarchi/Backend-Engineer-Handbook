## Исходная ситуация

Пользователь ввёл URL:

```text
https://company.com/api/files/123
```

Мы уже разобрали:

```text
✓ URL содержит host: company.com
✓ DNS может превратить company.com в IP-адрес
```

Допустим DNS вернул:

```text
company.com -> 203.0.113.17
```

Теперь браузер знает, куда нужно идти.

Но что вообще означает этот адрес?

Почему нельзя просто сказать:

```text
Отправь запрос в company.com
```

Почему нужен именно IP?

---

## Зачем нужна эта глава

Backend-разработчику не нужно быть сетевым инженером.

Но нужно понимать, что любой HTTP-запрос в итоге идёт не в контроллер напрямую, а к конкретному сетевому адресу.

IP-адрес важен при работе с:

| Где | Почему важно |
|---|---|
| Docker | контейнеры получают внутренние IP |
| Nginx | проксирует запросы на IP/host backend-сервиса |
| Kestrel | слушает конкретный address/port |
| PostgreSQL | клиент подключается к host/IP и port |
| Redis | подключение идёт к конкретному endpoint |
| RabbitMQ | сервисы подключаются к broker endpoint |
| Kubernetes | pod IP меняется, service name остаётся стабильным |
| Logs | часто видим client IP, forwarded IP, remote IP |

Если не понимать IP-адреса, трудно объяснять:

- почему `localhost` внутри контейнера — не твой компьютер;
- почему сервис доступен на `0.0.0.0`, но не на `127.0.0.1`;
- зачем нужны `X-Forwarded-For` и `Forwarded Headers`;
- почему Nginx может скрывать реальный IP клиента;
- почему один и тот же сервис может иметь разные адреса снаружи и внутри Docker-сети.

## Короткое определение

**IP-адрес (Internet Protocol address — адрес интернет-протокола)** — это сетевой адрес устройства или интерфейса в сети.

Он нужен, чтобы данные можно было доставить к нужному узлу.

Упрощённо:

```text
DNS отвечает: company.com = 203.0.113.17
IP отвечает: отправлять данные нужно к 203.0.113.17
```

Если DNS — это справочник имён, то IP — это конкретный адрес назначения.

---

## Простое объяснение

Представь город.

Название компании:

```text
Company LLC
```

само по себе не помогает курьеру доставить письмо.

Курьеру нужен адрес:

```text
ул. Примерная, дом 10
```

В интернете доменное имя похоже на название компании.

IP-адрес похож на адрес здания.

```text
company.com       → имя
203.0.113.17      → адрес
```

DNS помогает найти адрес по имени.

После этого уже можно пытаться установить соединение.

## IPv4 и IPv6

Чаще всего в backend-разработке встречаются два формата.

### IPv4

Пример:

```text
203.0.113.17
```

IPv4 состоит из четырёх чисел от `0` до `255`, разделённых точками.

Другие примеры:

```text
127.0.0.1
192.168.1.10
10.0.0.5
```

### IPv6

Пример:

```text
2001:db8::1
```

IPv6 появился, потому что IPv4-адресов стало недостаточно для глобального интернета.

Для Middle .NET backend-собеседования обычно достаточно понимать:

- IPv4 всё ещё очень часто встречается;
- IPv6 тоже существует и активно используется;
- современные приложения и инфраструктура могут работать с обоими вариантами;
- в конфигурациях и логах можно встретить оба формата.

Глубоко уходить в устройство IPv6 для обычного backend-интервью чаще всего не требуется.

## Что такое localhost

Один из самых важных адресов для разработчика:

```text
127.0.0.1
```

Обычно он означает:

```text
этот же компьютер
```

Имя:

```text
localhost
```

обычно преобразуется в loopback address (loopback-адрес — адрес, который указывает на эту же машину или контейнер), например:

```text
127.0.0.1
```

или IPv6-вариант:

```text
::1
```

Loopback нужен, чтобы приложение могло обращаться к сервису на этой же машине.

Пример:

```text
http://localhost:5000
```

означает:

```text
обратиться к приложению на этой же машине, порт 5000
```

---

## Почему localhost часто ломает Docker

В обычной разработке `localhost` кажется очевидным.

Но внутри Docker это частая ловушка.

Если приложение запущено в контейнере:

```text
DirectoryService container
```

и внутри него указано:

```text
Host=localhost;Port=5432
```

то приложение будет искать PostgreSQL внутри этого же контейнера.

Но PostgreSQL обычно запущен в другом контейнере:

```text
Postgres container
```

Поэтому правильно обращаться по имени сервиса:

```text
Host=postgres;Port=5432
```

Упрощённо:

```text
localhost внутри контейнера = сам контейнер
postgres внутри Docker-сети = контейнер PostgreSQL
```

Это одна из самых частых ошибок при настройке Docker Compose.

---

## 0.0.0.0 и 127.0.0.1

Ещё одна важная разница.

### 127.0.0.1

Означает:

```text
слушать только локальные подключения
```

Если приложение слушает только `127.0.0.1`, оно доступно только внутри этой машины или контейнера.

### 0.0.0.0

Означает:

```text
слушать на всех доступных сетевых интерфейсах
```

В Docker это особенно важно.

Если ASP.NET Core приложение внутри контейнера слушает только `localhost`, снаружи контейнера до него может быть невозможно достучаться.

Часто контейнерное приложение должно слушать:

```text
http://0.0.0.0:8080
```

или через переменную окружения:

```text
ASPNETCORE_URLS=http://+:8080
```

Смысл:

```text
принимать подключения не только с loopback-интерфейса
```

## IP и Port работают вместе

IP сам по себе указывает на узел.

Но на одном узле может работать много приложений.

Например:

```text
203.0.113.17:443   -> HTTPS API
203.0.113.17:5432  -> PostgreSQL
203.0.113.17:6379  -> Redis
```

Поэтому для подключения почти всегда нужна пара:

```text
IP + Port
```

или:

```text
Host + Port
```

Пример:

```text
company.com:443
postgres:5432
redis:6379
rabbitmq:5672
```

IP отвечает на вопрос:

> К какой машине идти?

Port отвечает на вопрос:

> В какую дверь на этой машине постучаться?

Порт подробно разберём в следующей главе.

---

## IP в ASP.NET Core и Kestrel

ASP.NET Core приложение обычно принимает запросы через Kestrel.

Kestrel должен слушать конкретный endpoint.

Например:

```text
http://localhost:5000
```

или:

```text
http://0.0.0.0:8080
```

В `launchSettings.json`, Dockerfile, `docker-compose.yml` или переменных окружения можно встретить настройки, которые определяют, где приложение слушает входящие подключения.

Пример:

```yaml
environment:
  ASPNETCORE_URLS: http://+:8080
```

Это важно, потому что приложение может быть запущено корректно, но быть недоступно из-за неправильной настройки адреса, на котором оно слушает подключения.

---

## IP клиента и Reverse Proxy

Когда приложение стоит за Nginx, оно часто видит не реальный IP клиента, а IP обратного прокси.

Упрощённо:

```text
Client -> Nginx -> ASP.NET Core
```

Для ASP.NET Core непосредственный клиент — это Nginx.

Поэтому `RemoteIpAddress` может показывать адрес Nginx, а не реального пользователя.

Чтобы передавать реальный IP, используют специальные headers, например:

```text
X-Forwarded-For
X-Real-IP
```

В ASP.NET Core для корректной обработки таких заголовков используют Forwarded Headers Middleware.

Эта тема будет подробнее в главах про Nginx и ASP.NET Core pipeline.

Сейчас важно понять базовую мысль:

> IP, который видит backend, не всегда равен реальному IP пользователя.

---

## Практика из проекта

В Docker-инфраструктуре backend-сервисы обычно не должны обращаться друг к другу по случайным IP контейнеров.

IP контейнера может измениться при пересоздании.

Поэтому сервисы обращаются по имени:

```text
directory-service
file-service
auth-service
postgres
redis
rabbitmq
```

Например:

```text
Host=postgres;Port=5432
```

или:

```nginx
proxy_pass http://directory-service:8080;
```

Здесь IP остаётся внутренней деталью Docker-сети.

Сервис работает с именем, а инфраструктура находит для него актуальный IP.

Это снижает связанность между сервисами и конкретными сетевыми адресами.

## Типичные ошибки

### Ошибка 1. Думать, что IP и DNS — одно и то же

DNS — это механизм поиска адреса по имени.

IP — это сам сетевой адрес.

```text
company.com      -> имя
203.0.113.17     -> IP
```

---

### Ошибка 2. Использовать `localhost` внутри контейнера для доступа к другому контейнеру

`localhost` внутри контейнера указывает на сам контейнер.

Для другого сервиса в Docker Compose обычно нужно использовать имя сервиса:

```text
postgres
redis
rabbitmq
```

---

### Ошибка 3. Слушать только `127.0.0.1` внутри контейнера

Если приложение внутри контейнера слушает только loopback, оно может быть недоступно снаружи контейнера.

Для контейнерного приложения часто нужно слушать все интерфейсы:

```text
0.0.0.0
```

или:

```text
http://+:8080
```

---

### Ошибка 4. Думать, что backend всегда видит реальный IP пользователя

Если перед приложением стоит Nginx, Load Balancer или API Gateway, приложение может видеть IP обратного прокси.

Для получения реального IP нужны Forwarded headers (проброшенные заголовки — заголовки, через которые прокси передаёт исходные параметры запроса) и корректная настройка доверенных прокси-серверов.

## Когда это особенно важно

IP-адреса особенно важны при работе с:

- Docker Compose;
- Kestrel;
- Nginx;
- Kubernetes;
- PostgreSQL connection strings;
- Redis/RabbitMQ endpoints;
- логированием IP клиента;
- rate limiting по IP;
- security audit;
- обратным прокси как входным слоем;
- настройкой HTTPS и redirect URI.

<summary>Ответ</summary>

### Middle: Почему `localhost` внутри Docker-контейнера часто приводит к ошибкам?

<details>
<summary>Ответ</summary>

### Senior: Почему backend за Nginx может видеть не реальный IP пользователя?

<details>
<summary>Ответ</summary>

Потому что для ASP.NET Core непосредственным клиентом является Nginx или другой обратный прокси. Поэтому `RemoteIpAddress` может показывать IP обратного прокси.

## Ответ для собеседования

IP-адрес — это сетевой адрес узла или интерфейса, который нужен для доставки данных по сети. DNS помогает получить IP по имени, но само соединение устанавливается уже к IP и порту. Для backend-разработчика важно понимать IP в контексте Docker, Kestrel, Nginx и connection strings. Например, `localhost` внутри контейнера указывает на сам контейнер, поэтому для подключения к PostgreSQL в Docker Compose обычно используют имя сервиса `postgres`, а не `localhost`. Также важно помнить, что приложение за обратным прокси может видеть IP прокси-сервера, а не реального клиента; для этого используются forwarded headers.

---

## Шпаргалка

- IP — сетевой адрес узла.
- DNS превращает имя в IP.
- `127.0.0.1` / `localhost` обычно означает текущую машину или текущий контейнер.
- `0.0.0.0` означает слушать все доступные сетевые интерфейсы.
- В Docker лучше обращаться к сервисам по имени, а не по IP.
- IP контейнера может измениться при пересоздании.
- IP без port обычно недостаточен для подключения к приложению.
- Backend за Nginx может видеть IP прокси-сервера, а не пользователя.
- Для реального IP используют forwarded headers.
